# URL Shortener API

A REST API for creating and resolving shortened URLs, built as Project 1 of a self-directed backend engineering curriculum.

## What It Does

- Shortens long URLs into compact, unique slugs
- Redirects visitors from a short link to the original URL
- Tracks click counts per link
- Lists all created URLs with cursor-based pagination
- Supports deleting a shortened link

## Tech Stack

- **Framework:** FastAPI
- **Database:** PostgreSQL
- **ORM:** SQLAlchemy 2.0
- **Migrations:** Alembic
- **Dependency management:** uv
- **Containerization:** Docker Compose

## Architecture
Client → FastAPI routes → crud.py (business logic) → SQLAlchemy models → PostgreSQL
- `config.py` — loads and validates environment configuration
- `database.py` — SQLAlchemy engine/session setup, injected via FastAPI's dependency system
- `models.py` — the `urls` table definition
- `schemas.py` — request/response validation, separate from the database model
- `crud.py` — all database operations (create, read, update, delete, paginate)
- `utils/slug_generator.py` — generates unique slugs via a Postgres sequence + base62 encoding
- `routers/urls.py` — HTTP route handlers

## Slug Generation Strategy

Slugs are generated by reserving the next value from the `urls` table's underlying Postgres sequence, then encoding that integer using base62. This guarantees uniqueness by construction — no collision handling is required, since every reserved sequence value is different by definition.

## API Endpoints

| Method | Path              | Description                          |
|--------|-------------------|--------------------------------------|
| POST   | `/shorten`        | Create a shortened URL               |
| GET    | `/{slug}`         | Redirect to the original URL         |
| GET    | `/stats/{slug}`   | View click stats for a URL           |
| DELETE | `/{slug}`         | Delete a shortened URL               |
| GET    | `/urls`           | List URLs (cursor-based pagination)  |

## Running Locally

1. Clone the repo
2. Copy `.env.example` to `.env` and fill in your database credentials
3. Start Postgres: `docker-compose up -d`
4. Install dependencies: `uv sync`
5. Run migrations: `uv run alembic upgrade head`
6. Start the app: `uv run uvicorn app.main:app --reload`
7. Visit `http://127.0.0.1:8000/docs` for interactive API documentation

## Known Issues / Technical Debt

### `expires_at` column type (VARCHAR instead of TIMESTAMP)

While building out the routes, a migration issue led to converting the
`expires_at` column from a timestamp type to `VARCHAR`. This was a
reactive fix to unblock a migration error rather than a deliberate
design choice, and it comes with real tradeoffs:

- **What still works:** existing values remain valid ISO 8601 strings,
  and reading/writing expiry dates through the API continues to work
  as long as the format stays consistent.
- **What's broken as a result:** SQL-level date comparisons
  (e.g. `WHERE expires_at < NOW()`) won't work correctly against a
  string column. Sorting by expiry date isn't guaranteed to match
  chronological order. Schema-level validation of the field is weaker,
  since Pydantic can't enforce "this must be a real datetime" as
  strictly against a string-backed column.
- **What this blocks going forward:** any feature involving expiry
  logic — auto-deleting expired links, filtering "active" vs.
  "expired" URLs, sorting by expiry — will require fixing this first.
  The correct fix is a new Alembic migration that casts the columnsssss
  back to `TIMESTAMP WITH TIME ZONE` using an explicit `USING`
  clause, handling existing NULL values gracefully.

This is being left as-is for now since Project 1 doesn't currently
implement expiry-based logic, but it's flagged here as debt to
revisit before any such feature is built.
